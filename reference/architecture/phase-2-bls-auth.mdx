<Note>
  🔨 **Planned for T1.** Replaces the pre-authorization relayer with
  on-chain BLS aggregate verification and lands several defensive
  contract changes in the same cutover.
</Note>

Phase 2 removes the largest live trust assumption in Phase 1 — the
manager-signing relayer — and replaces it with a stateless
authorization model where the two counterparties on a trade are the
only signers gating their own settlement.

## Delta from Phase 1

What changes vs [Phase 1](/reference/architecture/current-state):

<CardGroup cols={2}>
  <Card title="Pre-auth gone" icon="circle-xmark">
    `_consumeAuth`, `managers[]`, `requestTokens[]`, and the
    `signature / authToken / timeToExpire` parameters are removed
    from every state-changing entry point on both `AdManager` and
    `OrderPortal`.
  </Card>
  <Card title="BLS verification on `unlock`" icon="key">
    Each `unlock` does one native pairing check
    (`env.crypto().bls12_381().pairing_check` on Soroban,
    `BLS12_PAIRING_CHECK` precompile on EVM) against the aggregated
    maker + bridger signature.
  </Card>
  <Card title="`BLSKeyRegistry`" icon="address-book">
    New contract. Maps each wallet to a registered BLS public key,
    with Proof-of-Possession enforced at registration to prevent
    rogue-key attacks. Single key per address in T1; a multi-key
    upgrade lands in Phase 3.
  </Card>
  <Card title="Route commitment per ad" icon="lock">
    Every ad pins its `committed_order_token` at creation time.
    Future admin route changes don't reach into existing ads —
    closes the admin-compromise → fake-token-route drain vector.
  </Card>
  <Card title="Side-bound MMR leaves" icon="layer-group">
    `MerkleManager` leaves are now `poseidon2(orderHash, sideFlag)`,
    closing the X-1 attack surface where a local-chain leaf could
    satisfy a counterpart-chain proof.
  </Card>
  <Card title="Pause + 2-step admin" icon="pause">
    `whenNotPaused` guard on every state-changing entry point.
    Two-step admin transfer (`AccessControlDefaultAdminRules` on
    EVM, `pending_admin / accept_admin` on Soroban) eliminates the
    wrong-address transfer footgun.
  </Card>
</CardGroup>

## What gets signed

Both parties sign the **OrderData struct** — the same 15-field
EIP-712 Order from Phase 1. No additional fields, no domain-tag
prefix, no per-chain variation. The orderHash is byte-identical on
both chains by construction, so a single aggregated signature
verifies on either side:

```text
authMessage  = orderHash                     // EIP-712 digest of the OrderData struct
sigAggregate = sigMaker + sigBridger         // G1 point addition
pkAggregate  = pkMaker  + pkBridger          // G2 point addition

// Pairing check on each chain's `unlock`:
e(sigAggregate, G2_generator) == e(H_to_G1(orderHash), pkAggregate)
```

Total signatures per trade: **2** (one per user). Users sign once;
the aggregate is reused at both `AdManager.unlock` and
`OrderPortal.unlock`.

## Curves at a glance

| Concern | Curve | Why |
|---|---|---|
| BLS signature aggregation + on-chain verification | BLS12-381 | Native Stellar host functions (CAP-0059), native EVM precompile (EIP-2537), 128-bit security, matches the Noir BLS library |
| Deposit ZK proof (MMR inclusion + nullifier) | BN254 | UltraHonk native; auto-generated verifier (Soroban + Solidity) already deployed in Phase 1 |

The two curves don't compose at the contract layer in T1 — the
deposit proof and the BLS verification are independent gates on
`unlock`. A combined proof is a long-term optimization tracked in
[Phase 5](/reference/architecture/phase-5-followups).

A standalone Noir BLS auth circuit also ships in T1 as a building
block — used in later phases for agent attestations and dispute
evidence, not on the unlock path itself.
